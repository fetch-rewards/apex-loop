Apex Loop

A self-improvement loop for AI agents: it watches an agent in production, finds where it falls short, rewrites the agent's own instructions, proves the change is genuinely better with a real experiment, and ships only what a human approves — so agent quality is maintained by a loop, not by a person per agent.

01What a self-improvement loop is

In one line: Apex Loop automates the maintenance cycle a human engineer does by hand — find a failure, fix the agent's prompt, prove the fix helped, ship it — while keeping the human at the one decision that matters: what actually ships.

The idea, from first principles

An AI agent doesn't stay good on its own. The world shifts, prompts age, new edge cases appear, and quality quietly drifts. Today, keeping one agent healthy is manual: a person reads production transcripts, guesses at a prompt change, and hopes it helped without breaking something else.

A self-improvement loop turns that manual cycle into a repeating, mostly-automated one:

observe → find the failure → change the agent → verify the change is real → (human decides) ship → repeat.

The guiding principle throughout: humans decide; machines do everything up to the decision. It is automation of the improvement work, not blind autonomy — the system does the toil and brings a human the verified evidence to approve or reject.

What, precisely, it improves

To be exact about the claim: Apex Loop improves an agent by rewriting one component of its harness — the agent's prompt — verified against real evaluations, with the loop architecture built to extend to other harness levers (tools, code) later. It does not retrain the model's weights, and it does not attempt to fix the broader application or infrastructure — where it detects such a cause, it routes that to a human rather than acting.

Where it sits in the field

This is not a one-off. Through 2026 the field converged on a layered view of how agents are optimized — and named the loop layer explicitly (harness engineering, loop engineering). Each layer optimizes a wider unit of the system than the one below it:

Prompt engineering the words in a single model call
Context engineering what the model sees on a run (retrieval, state)
Harness engineering the scaffolding around a run: tools, memory, planning, evals, permissions
Loop engineering the cycle over time — how the agent iterates, verifies, escalates, and improves the layers below it across runs  ← Apex Loop

The field's central finding is the one Apex Loop is built around: a self-improvement loop is only as good as its verifier. Without a fast, trustworthy signal, a loop optimizes whatever proxy it is handed and drifts into confident regression. Every serious approach hard-gates the loop on a verifier — and the quality of that verifier, not the loop itself, is the real differentiator.

Field anchors: Weng, "Harness Engineering for Self-Improvement" (Jul 2026); the "Self-Harness" and "Agentic Harness Engineering" papers (2026); the 2026 loop-engineering write-ups. Independent third-party validation that the loop, gated by a verifier, is where the field is heading.

02Why it matters — the value

Primary value — scale: as Fetch runs more agents, the bottleneck becomes a human babysitting each one's quality. A single loop maintains many agents, so agent maintenance scales without scaling headcount — with humans only at the ship decision.

The problem, quantified by the direction of travel

Fetch is standing up agents across products, dev tooling, and operations. Each deployed agent needs ongoing quality maintenance. Done by hand, that is one skilled human per agent, forever — a cost that grows linearly with the number of agents. The loop breaks that coupling: the same machinery watches, diagnoses, and proposes fixes for any agent, and a human spends their time on the decision, not the detective work.

Supporting value

A reusable trust layer

The verifier machinery — scorer-validation, human-label ground truth, significance testing, an absolute safety floor — is reusable infrastructure any agent or eval effort at Fetch can build on, not a one-product tool.

Catches what manual review misses

A statistical, no-regression gate with a safety floor catches subtle regressions and unsafe changes before they ship — more reliably, and faster, than a person eyeballing transcripts.

This value framing maps directly onto how Innovation Week scores Impact — "can it scale to a major, widespread pain point for multiple teams" — and, as Section 5 shows, the need is already visible across several teams' roadmaps.

03The loop — six blocks around a dataset

The loop is six blocks driven by a deterministic orchestrator that halts at a human gate. The datasets (a frozen golden set, a growing learning set, a scorer-validation set) sit at the center; guardrails hold each step to "generalize, don't overfit."

The ring: observe (Capture→Signal) → decide (Diagnose) → act (Improve) → verify (Prove) → human gate (Promote), around a persistent dataset center. It repeats.

We ran the full loop on the hardest instantiation: the consumer shopping assistant, whose prompt is already heavily hand-tuned, on a fuzzy (LLM-judge) signal — the case that leans hardest on the trust machinery.

Experiment detail — beating a genuine older prompt agent-prompt lever · significant

What changed: the loop diagnosed UX Quality as the weakest bucket (0.785 across 497 human-labeled traces) and rewrote the base prompt's output contract to distill it — leading every answer with the conclusion, tightening the "don't restate raw tool output" rules, and cutting filler. The rewrite was shorter (222 → 216 lines), not padded — the change was sharper guidance, not more of it.

Set-up: baseline = a genuine older prompt version · candidate = the distilled rewrite · 152 matched item-pairs, 2 prove-runs.

Gate outcome: quality +0.0404 (95% CI [+0.0090, +0.0621], P(Δ>0)=0.996 — significant); safety floor did not hold (0.8412 → 0.8431, still under the 0.85 floor); variance not OK → is_improvement = False. A real gain, correctly not shipped (see Finding 2).

⛔ Quality rose significantly — but the candidate's safety composite (0.843) landed below the 0.85 floor, so the loop held the gate. Gain is real; safety is never traded for it.

Finding 2 — it refused to ship it HELD AT GATE

Verdict: not promoted. The safety composite (0.843) sat just under the 0.85 floor, driven by one near-ceiling safety judge wobbling on the sample. Quality was significantly up; the loop still declined, because safety is a floor that a quality gain can never buy back. A loop that just chased the quality number would have shipped this. Refusing is the loop working as designed — this is what makes it safe to put near production.

Finding 3 — self-authored prompt-only rewrites hit a ceiling

The significant gain above was measured against a genuine older version of the prompt. When the loop diagnosed from current production and authored its own rewrites, results landed within noise of zero, improving only after the generator was strengthened to distill rather than add to the prompt (measured 152-pair deltas moved from clearly negative toward ~+0.04, significant on the best runs). The reading: on an already-tuned prompt, prompt-only rewriting has a ceiling — the detection and proof machinery is solid; the generator is the frontier, and deeper wins need levers beyond the prompt.

Experiment detail — the loop rewriting a prompt three times, unaided agent-prompt lever · not significant

What changed: diagnosing from current production, the loop targeted the Tool Compliance bucket and authored its own rewrite of the base prompt (additive, +2 lines) to sharpen when and how tools must be called. It then looped three times — diagnose → rewrite → prove, diagnose → rewrite → prove again — each iteration trying to clear the bar.

Set-up: baseline = current production prompt · candidate = the loop's self-authored rewrite · 3 improve/prove iterations.

Gate outcome: across iterations the quality delta stayed within noise of zero (final aggregate −0.0113, 95% CI [−0.1024, +0.2298] — not significant); safety floor held, but variance not OK → is_improvement = False, rejected. The machinery ran flawlessly; the generator couldn't clear an already-tuned prompt unaided.

Finding 4 — it declined to rewrite when the problem was the judge, not the prompt DID NOT ACT

In one run the top failure bucket carried a low score on a thin, unvalidated sample, and the signal came back not trustworthy. The loop routed the fix to the scorer lever — meaning "the judge is the suspect here" — and skipped the Improve block entirely rather than author a prompt rewrite to paper over a shaky score. No candidate was produced; the run took no action. This is the counterpart to Finding 2: there the loop refused to ship a proven-unsafe change; here it refused to attempt a change at all when the evidence wasn't sound. Both are the same fail-closed reflex — the loop would rather do nothing than act on a signal it doesn't trust.

Experiment detail — declining to act on an untrustworthy signal scorer lever · no action

What changed: nothing — and that is the result. The top failure bucket ("Sensitive Topics & Appropriate Disclaimers") scored low (0.200) but on a single, thin sample, and scorer-validation flagged the signal as not trustworthy. The loop routed the fix to the scorer lever — "the judge is the suspect, not the prompt" — and skipped the Improve block entirely. No candidate was written, no prompt was touched.

Set-up: Capture → Signal → Diagnose ran; Improve was skipped by the lever routing before any rewrite was attempted.

Gate outcome: final_action = none, is_improvement = n/a (never reached Prove). The fail-closed counterpart to Finding 2: there it refused to ship a proven-unsafe change; here it refused to attempt one on evidence it couldn't trust.

05How it fits the broader landscape

The loop does not stand alone. Across Fetch, several efforts are each building a different layer of the "agents in production" stack; the loop's relationship to each is distinct. Named below by the effort/tool doing the work.

≈ MIRROR — same loop, elsewhereOther self-improving agentsincident RCA (devops-agent), coding agent (Buck Bronson), async matching, mobile PR pipeline
▼ CONSUME — uses our trust layerThe eval & skills infraLLM-judge eval + calibration (Opik, JADE) and the skills registry (Fetch Skills)

ENABLE — we run on it

Violet, the shared meta-harness, is building the one agent runtime the org runs on (sandboxes, orchestration, swappable models). That is the harness our loop runs on — and Violet's own roadmap lists "self-healing workflows" as an open dependency. The loop is exactly that missing piece: complementary, filling a stated gap, not competing. The fetch-agent-mesh control plane (with its own agent dev→test→promote→rollback lifecycle) and a standalone eval-harness primitive could likewise host the loop's Signal/Prove stages org-wide.

EMBED — we are a stage inside it

AgentMS, the agentic product-development mesh, runs the delivery lifecycle (idea → PRD → spec → PR) but ends at PR delivery — it has no post-deploy self-improvement stage. The loop is the continuous "improve + verify" step that closes that lifecycle from ship back to learn; and every agent in AgentMS (PM, researcher, tech-lead, spec-writer) is itself a candidate subject for the loop.

MIRROR — the same loop shape elsewhere

Independently, other efforts run the identical pattern on different signals — strong evidence the loop generalizes beyond our three planned use cases:

  • devops-agent (autonomous RCA) runs capture → diagnose → propose-fix → gate → earn-trust on incidents, growing its own eval set from bad runs — our loop's shape in a different domain, on a crisp signal.
  • Buck Bronson (and the mobile agentic-PR pipeline) learns from PR outcomes (tests, CI, review) — the same loop where an objective signal permits near-automatic promotion. This is one of Apex Loop's own three proving use cases.
  • An async matching optimizer improves its prompt on a slow cadence with a crisp accept/reject signal — the loop with the trust machinery relaxed because ground truth is cheap (a second of the three use cases).

CONSUME — uses the trust layer we build on

The Opik-based LLM-judge evaluation system (judges, composites, a golden set, and JADE judge-vs-human calibration) is the substrate the loop's Signal block is built from; an org eval-tooling selection effort and the Fetch Skills registry (which graduates skills only with eval evidence) both need exactly the trustworthy-verdict discipline the loop asserts. The company experimentation platform (A/B + holdouts + an emerging experiment-intake agent) is the product-scale sibling of the loop's Prove stage — and could serve it.

Note on this section. Violet's "self-healing workflows" gap is a confirmed open dependency. But the loop-as-a-stage-in-AgentMS and the Buck Bronson learning loop are proposed / in-progress, not yet operating systems, and the Opik/JADE eval substrate sits partly within our own team's orbit rather than being independent cross-team validation. The relationships are real; several are forward-looking rather than shipped.

06Limits & what's next

Stated plainly, at the level of the general loop (each use case hits these differently):

Limit (today)What it meansNext
Autonomous rewrite isn't a clean win yet The loop's own prompt rewrites improve directionally but don't yet clear the bar unaided on an already-tuned prompt. The detection + proof works; the generator is the weak link. A stronger generator (candidate-panel / search-based refinement) and broader change surfaces than the prompt.
Prompt-only scope this week The change surface was bound to the prompt; tools, code, and the harness itself were out of scope. Extend the same loop to the other harness levers (the architecture already generalizes).
Safety margin is noise-sensitive at demo sample sizes The floor tripped on a near-ceiling judge's wobble, not a real regression — correct, but the margin moves on small samples. More proof repeats / a larger safety sample to stabilize the floor.
The signal rests on the judges — and the agent is non-deterministic Every verdict inherits the judges' own calibration state; "13/13 validated" means validated against a limited set of human labels, not proven correct in the absolute. And the agent produces different outputs run-to-run, so any single run is a noisy sample — the safety judge's wobble was exactly this. Keep re-validating judges as human labels grow; lean on significance + repeats rather than single runs; treat judge drift as its own signal to fix the judge.
Why the loop is built the way it is. The two uncertainties above — judges that may be miscalibrated, and an agent whose behavior is random run-to-run — are not after-thoughts; they are the reason for the whole trust apparatus. Scorer-validation exists because a judge can be wrong. The paired significance test and the repeats exist because one run is noise. The safety floor exists because a single near-ceiling judge can wobble a sample. A naive loop that trusted one judge on one run would have shipped the rewrite in Finding 2. Ours did not — the machinery is a direct response to a shaky foundation, not a claim that the foundation is solid.
A field caveat worth carrying: the literature warns that harness churn is not harness benefit — a loop that changes things is not the same as a loop that improves them. This is precisely why the loop's value lives in the verifier and the gate, not in the rewriting. We live this distinction: our own rewrites came out neutral until proven, and were correctly not shipped.
